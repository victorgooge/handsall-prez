Testing Strategy
Testing Methodology & QA Deployment
The complete testing stack: dashboard entry point, emulator tools, GPS simulation, and physical-device beta validation
Old vs. New Testing Approach
  • Previous: Custom test harness + Android Virtual Device location simulation
  • Limitations: No real-time location tracking, high latency, tedious to run
  • Now: Physical device testing with campus-based real locations (GSU)
  • Advantage: Realistic user flows, accurate GPS behavior, repeatable test environment
Private "Testing" Category
  • Dedicated category in app UI (not a separate harness)
  • Routes testers to GSU campus locations
  • Full flow: category selection → activity → clues → destination
  • Mirrors production behavior exactly
  • Accessible locations for interns (campus buildings, libraries, plazas)
Team Coordination & Ownership
  • Victor (Frontend): GitHub org access, testing guidelines, final flow demos
  • Ike (Backend): "Testing" category backend setup, tester role assignment
  • QA Team (Hema, Eka, Viviana): TestFlight deployment, device testing, pass/fail reporting
  • Weekly sync on blockers, feature readiness, and test coverage
Testing Infrastructure & Tools
  • Dashboard entry: QA starts inside the real Zoku app through the private Testing category.
  • Location control: Dev tools load event coordinates and compute spawn, target zone, and destination points.
  • Route playback: Emulator controls still support repeatable route simulation for quick checks.
  • Physical-device validation: TestFlight testing confirms GPS accuracy, clue progression, and Proximity Search behavior.
